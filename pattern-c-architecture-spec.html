Summary

The corrected trust boundary

PR autofix is an adapter over reusable platform contracts. GitHub events enter a durable workflow; the trusted worker prepares a credential-free workspace and gives an untrusted sandbox only that workspace at /work plus an opaque, short-lived capability token.

The sandbox cannot read upstream credentials, access Docker, or choose a repository destination. It edits and tests local files. A trusted capability broker relays Anthropic messages, exposes minimal read-only PR MCP tools, and forwards a request to publish as a Temporal Workflow Update. Only after an authorized human comment does a trusted activity commit and push once.

Components

Trusted and untrusted responsibilities

Gateway

Verifies webhook HMAC, enforces ALLOWED_REPOS, starts workflow runs, and accepts approval/denial issue comments only from APPROVER_LOGINS.

Temporal

Persists workflow state, pending operations, decisions and outcomes. Local Compose mounts temporal-dynamicconfig.yaml to enable Workflow Updates.

Worker

Prepares `${WORKSPACE_ROOT}` clones with a clean origin, provisions sandboxes, executes trusted status/push activities, and owns Docker socket access.

Capability broker

Registers opaque run bindings, relays allowed Anthropic Messages paths with the trusted key, and exposes scoped remote HTTP MCP tools.

Sandbox

Has /work only, cap_drop=ALL, no-new-privileges and resource/network limits. It runs agent edits and repository tests under containment.

Trusted push

Stages and commits from the workspace after approval, pushes only the workflow-bound branch, and stamps an idempotency trailer.

Sequence

End-to-end PR autofix path

#ActionControl
1GitHub sends a PR or review-comment webhook.Gateway verifies HMAC and repository allowlist.
2Workflow prepares the trusted workspace.Temporary Git Basic header is used; persisted origin contains no credential.
3Worker registers a capability token and starts the sandbox.Only the individual workspace is mounted at /work; no Docker socket or upstream token.
4Agent reads context, edits and tests.Model and GitHub reads flow through scoped broker endpoints.
5Agent requests publication.Temporal Update records a deterministic pending operation and posts an approval comment.
6Reviewer replies /autofix approve <approval_id>.Gateway accepts configured approver logins only; denial remains possible.
7Trusted activity commits and pushes.Destination comes from PRRef; commit contains Autofix-Idempotency: <key>.
8Workflow posts status and tears down.A repeated publish request returns the recorded result rather than producing another commit.

Live verification. A real PR review comment triggered an isolated edit, generated a pending approval operation, was approved by GitHub comment, and produced one pushed commit. Retrying the publish request returned the same commit SHA.
PoC authentication choice. Trusted services currently use a fine-grained GitHub PAT. Production deployment should use GitHub App installation tokens. No PAT is provided to the sandbox.
Operations

Configuration and reproduction

Trusted service configuration in .env:

GITHUB_TOKEN=<trusted-service PAT>
ANTHROPIC_API_KEY=<trusted broker key>
GITHUB_WEBHOOK_SECRET=<signature secret>
ALLOWED_REPOS=owner/repo
APPROVER_LOGINS=authorized-login
WORKSPACE_ROOT=/tmp/agent-temporal-workspaces
BROKER_REGISTRATION_SECRET=<random secret>

Sandbox-generated values are capabilities and routes, not upstream credentials:

RUN_CAPABILITY_TOKEN=<opaque short-lived token>
ANTHROPIC_BASE_URL=http://capability-broker:8443/anthropic
CAPABILITY_MCP_URL=http://capability-broker:8443/mcp
docker compose up -d --build --remove-orphans
uv run python scripts/trigger_webhook.py https://github.com/owner/repo/pull/123
# When an approval request is posted:
# /autofix approve <approval_id>

Local endpoints: Gateway http://localhost:8000/webhook, Temporal UI http://localhost:8233, Phoenix http://localhost:6006.
